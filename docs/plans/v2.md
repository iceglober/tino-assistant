# Tino v2 — Proactive Assistant

## Goal

Evolve tino from a reactive DM bot into a proactive assistant that can schedule its own work, message the owner unprompted, read Slack channel context, and manage email — all backed by BAA-compliant models on Bedrock.

## Constraints

- **BAA compliance.** All models must be BAA-backed (Bedrock). All data in transit is TLS. All data at rest is encrypted (Fargate ephemeral storage, SQLite on encrypted volume). Audit logging via pino with PHI redaction.
- **Single-user.** `ALLOWED_SLACK_USER_ID` remains the sole gate. Multi-user is not in scope.
- **Tool budget.** Lifted — no hard cap. Monitor tool-definition token overhead at startup; if it exceeds ~4K tokens, consider tool grouping or dynamic tool selection.
- **Deployment target.** ECS Fargate on the existing work AWS account.

## Items

### Phase 9 — Task primitive + proactive messaging
- [x] **9a.** SQLite `tasks` table: `{ id TEXT PK, user_id TEXT, description TEXT, scheduled_at INTEGER, status TEXT ('pending'|'running'|'completed'|'failed'|'cancelled'), result TEXT, created_at INTEGER, updated_at INTEGER }`.
- [x] **9b.** `schedule_task(description, scheduledAtIso)` tool — Claude can schedule future work.
- [x] **9c.** `list_tasks(status?)` tool — Claude can see pending/completed tasks.
- [x] **9d.** `cancel_task(taskId)` tool — Claude can cancel a pending task.
- [x] **9e.** Scheduler loop: `setInterval` (60s) queries `scheduled_at <= now AND status = 'pending'`, runs each through `runAgent` with the task description as input, posts result to the owner's DM proactively.
- [x] **9f.** Proactive DM: resolve the owner's DM channel ID at startup via `conversations.open`, cache it. Use `chat.postMessage` for scheduler-initiated messages (not `say`, which only works inside an event handler).
- [x] **9g.** Startup log: tool-definition token count estimate (count tokens in all tool descriptions + JSON schemas).
- [x] **9h.** Remove the ~10 tool cap from the system prompt.
- [x] **9i.** System prompt update: teach Claude about `schedule_task` and when to use it (e.g., "if you see a meeting tomorrow that needs prep, schedule a task for 1 hour before").

### Phase 10 — Slack thread reading (user token architecture)
- [x] **10a.** Design: user OAuth token (`xoxp-`) obtained via Slack OAuth flow with user scopes (`search:read`, `channels:history`, `groups:history`). Stored in `.env` as `SLACK_USER_TOKEN`. The bot reads channels *as the owner*, not as itself — sees exactly what the owner sees.
- [x] **10b.** `slack_search_messages(query, count?)` tool — wraps `search.messages` API (requires `search:read` user scope). Returns `{ messages: [{ channel, ts, text, user, permalink }] }`. This is the most powerful context-gathering tool — "what did the team discuss about X?" is one call.
- [x] **10c.** `slack_read_thread(channel, threadTs, limit?)` tool — wraps `conversations.replies` (requires `channels:history` or `groups:history` user scope). Returns the full thread. Use for "catch me up on this thread" or for the scheduler to read threads referenced in meeting prep.
- [x] **10d.** System prompt update: teach Claude about Slack reading tools and when to use them (meeting prep, task context gathering, catching up on missed discussions).
- [x] **10e.** Slack app config: add user scopes (`search:read`, `channels:history`, `groups:history`), re-install to workspace, obtain user token.

### Phase 11 — Autonomous background analysis
- [ ] **11a.** Morning briefing: cron-like trigger (configurable, default 7:00am owner's timezone) that runs a "daily briefing" agent pass — reviews today's calendar, yesterday's unread emails, recent CI failures, and proactively DMs a summary + schedules prep tasks for upcoming meetings.
- [ ] **11b.** Continuous monitoring (optional): periodic (every 30 min?) lightweight check — new CI failures, new emails from VIPs, calendar changes. Only DMs if something actionable is found. Configurable via preferences (`monitoring_interval`, `vip_senders`, etc.).
- [ ] **11c.** Task chaining: when a scheduled task completes, it can schedule follow-up tasks (e.g., "prep notes written → schedule a reminder 15 min before the meeting to review them").

### Phase 12 — Email management (write operations)
- [ ] **12a.** Expand Gmail scope from `gmail.readonly` to `gmail.modify`. Re-run auth script.
- [ ] **12b.** `gmail_trash_messages(messageIds[])` tool — move messages to trash. Requires confirmation flow (Claude must list the messages and ask "delete these?" before calling).
- [ ] **12c.** `gmail_label_messages(messageIds[], labelIds[])` tool — apply/remove labels. Useful for "mark these as read" or "label these as follow-up".
- [ ] **12d.** Confirmation flow for write operations: Claude must surface what it's about to do and wait for explicit "yes" before executing. This is the approval UI deferred from Phase 5, now load-bearing.

### Phase 13 — Deployment (ECS Fargate)
- [x] **13a.** Dockerfile: multi-stage build (install deps → copy src → `tsx src/index.ts`). No bundler needed.
- [x] **13b.** ECS task definition: 0.25 vCPU / 0.5 GB, secrets from Secrets Manager (Slack tokens, Google refresh token, GitHub PAT, AWS creds for Bedrock/CloudWatch).
- [x] **13c.** ECR repository for the container image.
- [x] **13d.** Pulumi (or CDK/Terraform) IaC for the ECS service, task definition, ECR repo, Secrets Manager secrets, IAM roles, VPC/subnet config.
- [x] **13e.** Deploy script: `docker build && docker push && aws ecs update-service --force-new-deployment`.
- [x] **13f.** SQLite on EFS (Elastic File System) — Fargate ephemeral storage is wiped on task restart. EFS mount gives persistent storage for `tino.db` across deploys. Alternative: migrate to DynamoDB or RDS, but SQLite-on-EFS is simpler for a single-process bot.
- [x] **13g.** Health check: ECS health check that verifies the Slack WebSocket is connected. If the connection drops, ECS restarts the task.
- [x] **13h.** Logging: pino output → CloudWatch Logs (stdout capture by ECS). Structured JSON logs, same redaction config as local.

### Phase 14 — Per-repo config system
- [x] **14a.** Design: SQLite `config` table in `tino.db` for system-wide runtime configuration. Keys use dot-notation namespacing (`github.repos`, `cloudwatch.log_groups`, etc.). Values are JSON strings. Editable via localhost web console at port 3001.
- [x] **14b.** Migrate hardcoded allowlists from `src/tools/github/allowlist.ts` and `src/tools/cloudwatch/allowlist.ts` to read from the config store at tool-construction time. `isAllowedRepo` and `getAllowedLogGroups` now take the allowlist as a parameter (no module-level constants).
- [x] **14c.** Config-driven tool registration: allowlists are resolved from ConfigStore at `buildTools` time. Empty allowlist → tool is registered but fails-closed on every call (same behavior as before, now runtime-configurable).
- [x] **14d.** Multi-repo support: `github.repos` is a JSON array of `"owner/repo"` strings. `github.default_repo` is a JSON string. The `GITHUB_DEFAULT_REPO` env var is kept as a fallback for backward compat.

## Open questions

- **Task execution context.** When the scheduler fires a task, should it run with the full conversation history (so Claude has context from when the task was scheduled), or with a fresh context (just the task description + system prompt)? Fresh is simpler and avoids stale-context bugs; full-history is more powerful but the history may have been trimmed or reset since scheduling.
- **Scheduler timezone.** Tasks are stored as UTC epoch seconds. Claude needs to know the owner's timezone to schedule "1 hour before my 10am meeting" correctly. The preferences system already supports this (`timezone` preference), but Claude needs to be told to check it.
- **Slack user token storage.** The user token (`xoxp-`) is more sensitive than the bot token — it has the owner's full channel access. Should it go in `.env` alongside the bot token, or in a separate secrets store?
- **EFS vs. DynamoDB for persistence.** SQLite-on-EFS is the simplest migration from local SQLite. DynamoDB is more "cloud-native" but requires rewriting the persistence layer. For a single-user bot, EFS is probably fine.
- **Confirmation flow design.** Phase 12's email deletion needs a "Claude asks, user approves" flow. This is the same problem as Phase 5's `needsApproval` — but now it's load-bearing. Design options: (a) AI SDK's built-in `needsApproval`, (b) custom Slack interactive message with buttons, (c) plain-text "reply approve/deny" like the original Phase 8 plan. Needs a decision before Phase 12.
